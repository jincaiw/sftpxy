# 手工模拟 + 全面测试 + 修复执行计划

## 目标

本轮按以下顺序执行：

1. 先计划
2. 启动应用
3. 模拟人操作
4. 进行一次全面测试
5. 修复发现的问题和 bug
6. 再次测试，确认回归通过

本计划只定义执行方式、覆盖范围、优先级和退出条件。在收到确认前，不进入实际执行阶段。

## 执行环境

- 启动方式优先使用 `tests/e2e/start-e2e.sh`
- 使用 `config.e2e.yaml` 作为测试配置
- 前端页面通过浏览器实际访问：
  - `http://127.0.0.1:30088/admin/login`
  - `http://127.0.0.1:30080/client/login`
- 测试账号优先使用 e2e 种子：
  - admin: `admin / admin-pass`
  - user: `testuser / testuser-pass`

## 范围

### 人工模拟路径

- 管理端登录
- 客户端登录
- 客户端文件列表浏览
- 基本上传 / 下载 / 删除
- 管理端连接列表查看
- MFA 相关页面与关键交互
- OIDC / 联邦流程入口可达性与现有回跳兼容性

### 自动化测试范围

- Go 全量测试
- 协议综合测试
- API 综合测试
- UI 综合测试
- Playwright E2E
- 生产配置检查

## 分阶段计划

### 阶段 1：环境准备与启动

- 确认依赖和构建状态可用
- 启动 e2e 服务
- 验证 `/health`、`/status`、登录页可访问
- 记录启动阶段异常

退出条件：

- 服务端正常监听
- 管理端和客户端页面可打开
- 种子账号可用于登录

### 阶段 2：人工模拟真实操作

- 使用浏览器逐步执行 admin 和 client 的真实登录流程
- 检查页面加载、导航、关键按钮和表单交互
- 对文件管理做最小闭环操作：
  - 查看目录
  - 上传文件
  - 下载文件
  - 删除文件
- 检查管理端连接列表与会话表现
- 对异常页面、控制台错误、网络异常做记录

退出条件：

- 关键主路径至少完成一轮
- 已记录所有可复现的 UI / 交互 / API 异常

### 阶段 3：全面自动化测试

按以下顺序执行：

1. `bash tests/protocol-comprehensive-test.sh`
2. `bash tests/api-comprehensive-test.sh`
3. `node tests/ui-comprehensive-test.js`
4. `npm --prefix web run test:e2e`
5. `go test ./internal/protocols/httpd -count=1`
6. `go test ./... -count=1`
7. `make test-prod-config`

执行原则：

- 每一类测试都记录失败点
- 对失败点做去重，区分真实回归、环境问题和历史已知问题
- 对高风险问题优先级排序：
  - 安全与认证
  - 数据正确性
  - 主流程不可用
  - UI 与兼容性

退出条件：

- 拿到完整失败清单
- 已完成问题归因和修复优先级划分

### 阶段 4：修复问题和 bug

- 优先修复可复现且影响主流程或安全性的真实问题
- 每修一类问题，优先补充或同步对应回归测试
- 对前后端联动问题一起修，不做半截修复
- 修复后立即检查最近修改文件的诊断与编译状态

退出条件：

- 已确认的真实问题全部修复
- 对应定向测试通过

### 阶段 5：再次测试

顺序如下：

1. 先跑受影响模块的定向测试
2. 再跑失败过的自动化测试
3. 再做一次人工主路径复测
4. 最后跑全量回归与生产配置检查

通过标准：

- 人工主路径通过
- 自动化失败项全部回归为通过
- Go 全量测试通过
- UI / API / 协议 / E2E 通过
- 不引入新的诊断错误

## 本轮重点关注风险

- 登录、MFA、恢复码、会话刷新、会话撤销
- `/api/v1/connections` 与 admin/user 会话一致性
- 文件上传下载删除与路径安全
- 分享能力与权限校验
- OIDC 登录回跳和前端接收逻辑
- 联邦用户关闭 MFA

## 交付产出

执行完成后输出以下结果：

- 修复清单
- 新增或调整的测试清单
- 最终验证结果
- 剩余风险说明
